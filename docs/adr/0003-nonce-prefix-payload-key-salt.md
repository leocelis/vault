# ADR-0003: Per-body-write nonce_prefix as the payload-key HKDF salt

- **Status:** Accepted
- **Date:** 2026-06-26
- **Deciders:** maintainers (Gate 0 G0.1)
- **Constraints:** C1, C8, C4 (via SC6)
- **Supersedes:** the payload-key derivation paragraph of [ADR-0002](0002-aead-xchacha20-poly1305-stream.md)
  (`payload_key = HKDF(ikm=data_key, salt=b"", info="vault-payload-v1")`). ADR-0002's cipher and
  STREAM decisions stand unchanged.

## Context

The original C1 derived one payload key per vault: `HKDF(ikm=data_key, salt=b"",
info="vault-payload-v1")`, documenting the empty salt as a safe deviation from age ("the data key
is unique per vault, making the nonce_prefix redundant"). That rationale covered **cross-vault**
key uniqueness but missed **cross-save** reuse within one vault: C4 deliberately never rotates the
data key, and STREAM chunk nonces are deterministic counters restarting at 0 on every encryption.
Two saves of the same vault therefore encrypted different plaintexts under identical
(key, nonce) sequences — XChaCha20 keystream reuse. A history-keeping sync backend (Git, Dropbox —
UC-07), or our own `vault.vlt.bak` (C32), could XOR corresponding chunks of two versions to
recover plaintext differences, and unchanged chunks were byte-identical across versions (an
edit-locality oracle). `master_seed` rotation did not help: it keys only the C10 block HMACs,
never the payload cipher.

## Decision

Restore age's construction. A 16-byte CSPRNG `nonce_prefix` lives in the plaintext header (C8,
covered by `header_hash`/`header_hmac`) and is the HKDF salt for the payload key:

```
payload_key = HKDF-SHA-256(ikm=data_key, salt=nonce_prefix, info="vault-payload-v1")
```

`nonce_prefix` MUST be regenerated on every save that writes the encrypted body, and MUST NOT be
regenerated by header-only rewrites (password rotation re-wraps a stanza without touching the
body — regenerating the salt without re-encrypting would orphan the stored ciphertext). SC6
records this binding and its priority (`C4 > C1_rotation_scope`).

## Alternatives rejected

- **Rotate the data key per save** — breaks C4's O(1) password rotation and forces re-wrapping
  every stanza (hardware factors included) on every save.
- **Persist the STREAM counter across saves** — fragile bookkeeping, fails exactly once
  (crash between body write and counter persist) and then reuses keystream silently.
- **Keep the empty salt, document the leak** — indefensible: plaintext-diff recovery against the
  stated adversary (untrusted storage, UC-07) defeats the product's core claim.

## Consequences

- Every body-writing save re-encrypts under a fresh payload key: no cross-version keystream or
  chunk-identity channel; ciphertext diff noise is total by design (UC-07 §3.1 documents the
  size/mtime channels that remain).
- One additional HKDF invocation on the open/save path (negligible).
- C1/C8 tests assert salt freshness across body-writing saves and salt stability across
  stanza-only rewrites.
